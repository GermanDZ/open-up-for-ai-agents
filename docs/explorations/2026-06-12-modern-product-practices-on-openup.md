# Exploration: Modern product-delivery practices layered on top of OpenUP

**Started:** 2026-06-12
**Question:** How do we introduce modern product-management and delivery
practices — value-based prioritization, per-feature outcome measurement,
feature-flagged rollouts, and multi-environment deployment — as a layer that
works **on top of** OpenUP, without modifying the base process?

## Context

Request from the project owner, with four concrete examples to cover:

1. **Project Manager → Product Manager**: prioritization driven by business
   value added, not just schedule/risk.
2. **Per-feature measurement**: every new feature ships with a way to measure
   impact, engagement, and returned value.
3. **Controlled rollouts**: feature flags and similar techniques to control
   feature exposure.
4. **Multi-environment deployment**: staging, beta, and production as
   first-class deployment targets.

Hard constraint: the layer must sit **on top** of OpenUP. The vendored
knowledge base (`docs-eng-process/openup-knowledge-base/`) is generated by the
converter and is read-only reference material; the framework already has the
precedence chain for layering — **framework rubric → project rules →
task-spec safeguards** ([project-config.md](../../docs-eng-process/project-config.md)),
where a layer may *add* criteria but never waive a base one. That is exactly
the shape this modernization should take.

## Notes

### What OpenUP already gives us, per theme

The good news: none of the four themes contradicts OpenUP — three of the four
have explicit anchors in the KB that the layer can cite and extend rather than
replace.

**1. Value-based prioritization — OpenUP already has the concept, our
framework doesn't surface it.**

- The KB embeds a **Product Owner** capability pattern, distinct from Project
  Manager: *"the one and only person responsible for managing the Product
  Backlog and ensuring the value of the work the development team performs"*,
  with explicit responsibilities to *"prioritize features according to market
  value"* and accountability for *"the profitability of the product (ROI)"*
  (`openup-knowledge-base/process/base/capabilitypatterns/product-owner.md`).
- Prioritization guidance already says stakeholders who pay *"have the final
  say on what capabilities to prioritize"*
  (`practice-management/iterative_dev/guidances/guidelines/prioritizing-work-items.md`),
  and a core OpenUP principle is *"balance competing priorities to maximize
  stakeholder value"*.
- **Gap in our layer:** `.claude-templates/teammates/` has `project-manager`
  but no product-facing role; the roadmap table (`ID | Title | Status |
  Priority | Depends on`) carries a bare high/medium/low priority with no
  business-value rationale; the **iteration-plan rubric has zero
  business-value criteria** (all 7 criteria are execution-focused). Only the
  vision rubric mentions value, at whole-project level.

**2. Per-feature outcome measurement — the biggest genuine gap.**

- KB `metrics.md` defines metrics as *"an interpretation of a measure … that
  you use to guide your project"* — but every example (defect backlog,
  velocity, coverage) is a *process/quality* metric. Iteration assessment asks
  *"did the release meet its functionality and quality goals?"* — pre-release
  questions only.
- The vision rubric requires ≥3 measurable success metrics, but **per
  project**, not per feature, and nothing closes the loop after deployment.
- **Gap:** nothing in the task-spec template, use-case rubric, or
  complete-task flow asks "how will we know this feature worked?" A feature
  can complete the entire lifecycle without ever defining impact, engagement,
  or value measures — let alone reading them back after release.

**3. Feature flags / rollout control — no KB anchor at all.**

- The deployment discipline covers plan, package, execute, verify, backout,
  and release communications, and the **Deployment Plan** artifact carries
  risks and success criteria — but exposure control is binary
  (deployed/backed-out). No canary, percentage rollout, or flag concepts
  anywhere in the KB. This is genuinely post-2012 practice.
- The natural OpenUP hook is that a flag is a *modern implementation of the
  backout plan*: "Develop Backout Plan" is already a core deployment task, and
  a kill-switch flag satisfies it more cheaply than a redeploy. Framing it
  that way keeps the layer "on top" instead of "instead of".

**4. Multi-environment deployment — KB has the lifecycle, not the topology.**

- The **Transition phase** explicitly includes *"beta test to validate that
  user expectations are met"* and *"running old and new systems in parallel"*
  — beta is already a sanctioned OpenUP stage, just not modeled as an
  *environment*.
- The **Release (hardening) iteration** concept and the Deployment Plan
  (*"timing and dependencies for deploying components to various nodes"*)
  gesture at topology but assume one production target. The environment
  discipline is about the *team's* tooling, not target infrastructure.
- **Gap:** no place in our templates/config to declare the environment chain
  (staging → beta → production), the promotion criteria between stages, or
  which stage a change is currently at.

### Where the layer plugs in (existing extension points)

| Extension point | Mechanism | Used for |
|---|---|---|
| `docs/project-config.yaml` `context:`/`rules:` | injected into every `/openup-create-*` skill | per-project env names, flag tooling, value scale |
| Rubrics (`.claude-templates/rubrics/`) | additive criteria, graded at assess/complete | new criteria: value statement, success measures, rollout strategy |
| Templates (`docs-eng-process/templates/`) | task-spec, iteration-plan, work-items-list, vision | new sections (Behavior measures, Rollout) |
| Teammates (`.claude-templates/teammates/`) | self-briefing role files | `product-manager` role |
| Skills (`.claude-templates/skills/`) | workflow gates | `/openup-transition`, `/openup-complete-task`, `/openup-start-iteration` touch-points |

All five are sanctioned "on top" surfaces — no KB file and no base-process
rule needs to change.

### Sketch of the concrete deltas (if promoted)

1. **Product Manager role**: add `teammates/product-manager.md`
   (+ compact variant), grounded in the KB Product Owner pattern. Owns the
   roadmap's value ordering; PM keeps orchestration/risk. Roadmap table gains
   a `Value` column (or a one-line value rationale per entry); start-iteration
   reads it when picking the next item. The existing stakeholder-decides rule
   is preserved — the product-manager role *represents* that voice.
2. **Success measures per feature**: task-spec template (and feature-level
   use-case template) gains a required `## Success Measures` section with
   three prompts — impact, engagement, returned value — each a concrete,
   checkable measure ("n/a" must be argued, mirroring the ambiguity-gate
   idiom from T-015). Task-spec rubric + iteration-plan rubric gain matching
   criteria. `/openup-complete-task` ticks that instrumentation for the
   measures exists, not just the feature code.
3. **Rollout strategy**: task-spec gains a `## Rollout` section (flagged?
   default state? kill-switch? removal task for the flag — flags are temp
   debt). Framed as the modern Backout Plan so it composes with the KB
   deployment tasks. A short guideline doc in `docs-eng-process/` covers
   flag hygiene (naming, expiry, cleanup task auto-added to roadmap).
4. **Environments**: `project-config.yaml` gains an `environments:` key
   (ordered list, e.g. staging → beta → production, with promotion criteria
   per hop). Deployment-plan/transition skills read it; `/openup-transition`
   maps its beta-test objective onto the `beta` environment instead of
   treating transition as a single hop to prod.

## Options Considered

- **Option A — "Modern product practice pack" inside `docs-eng-process/`**
  (framework change: templates + rubrics + new teammate + new config keys, as
  sketched above). Pro: every project bootstrapped from the template gets it;
  follows the exact precedent of Process v2 and the T-015…T-021 block; rubric
  criteria make the practices *graded*, not aspirational. Con: largest scope
  (~4 tasks, multi-surface), needs care to keep each addition additive so
  quick-track work isn't burdened (measures/rollout sections should be
  required on `standard`/`full` tracks, "n/a with reason" on `quick`).
- **Option B — Pure project-config layer** (ship only `project-config.yaml`
  `rules:` entries + a `docs/product/` guideline; zero framework edits). Pro:
  literally zero base-process change, deployable today in one project. Con:
  `rules:` can add rubric criteria but **cannot add roles, template sections,
  or config keys** — themes 1 and 4 don't fit; every project re-authors the
  rules; nothing is graded by default. Ruled out as the *whole* answer, but
  it's the right *pilot vehicle* for theme 2's wording before hardening it
  into the framework rubric.
- **Option C — Modify the vendored KB / base process docs** (e.g., rewrite
  the project-manager KB page as product-manager). Ruled out: KB is
  converter-generated reference material, and the request explicitly says
  "on top of OpenUP" — we extend via the precedence chain, we don't fork the
  base.

**Leaning: Option A**, with Option B's config layer used for the
project-specific parts (environment names, flag tooling, value scale) so the
framework stays project-agnostic.

## Open Questions

- Does `product-manager` **replace** `project-manager` in the default team, or
  sit alongside it? (KB treats Product Owner and PM as distinct; leaning
  *alongside*, with product-manager owning roadmap order and PM owning
  execution — replacing would change base-process role semantics.)
- What's the minimal value scale for prioritization that an agent can apply
  without inventing numbers — qualitative rationale per roadmap entry, or a
  scored column (e.g., value/effort)?
- Who/what *reads back* the success measures after release? A
  `/openup-measure-impact` skill in transition, a retro checklist item, or
  out of scope for the framework (project CI/analytics owns it)?
- Track interaction: are Success Measures and Rollout sections required on
  `quick` track, or `standard`+ only with "n/a — reason" allowed?
- Should `environments:` promotion criteria gate `/openup-complete-task`
  (done = in production?) or does "done" stay at "merged + deployed to first
  environment"?
