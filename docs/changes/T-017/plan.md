# T-017 — `/openup-next`: sequential continue-loop + derived `.openup/board.json` + Operations checkboxes

## Story

> **As an** agent (or an outer `/loop`/cron/human) driving OpenUP delivery one session at a time
> **I want** a single `/openup-next` skill that reads a deterministic, machine-derived execution board, claims the top READY collision-free lane, executes one cycle, ticks its progress, and exits through a legal exit
> **So that** "read the next task and execute" becomes one repeatable command whose queue is computed by the harness — not re-derived from prose roadmap notes each session — and two loops naturally pick disjoint lanes with no coordination beyond the repo.

INVEST check:
✅ Independent (builds only on shipped T-008/T-009 coordination + done T-015/T-016) · ✅ Negotiable (board fields, sort order, hat-tag syntax tunable) · ✅ Valuable (turns multi-step intake into one command; enables per-lane parallelism) · ✅ Estimable (1 session) · ✅ Small (one script + one skill + one template/convention change + policy edits) · ✅ Testable (board.py is deterministic → unit-testable against fixtures; cold-session dry-run of the skill)

## Requirements

1. `scripts/openup-board.py` deterministically generates `.openup/board.json` from active change-folder frontmatter (`docs/changes/*/plan.md`, excluding `archive/`), live leases (`<git-common-dir>/openup/claims/`), and Operations-checkbox state — invoking no model and using only the Python standard library.
2. Each board lane carries exactly: `task`, `title`, `track`, `state`, `lease`, `hat`, `next_action`, `plan`, `collides_with`, `depends_ok`.
3. Lane `state` is derived deterministically: `in-progress` (live lease or frontmatter `in-progress`), `ready` (status `proposed`/`ready`, deps satisfied, no collision, no lease), `blocked` (unmet dependency), `colliding` (deps ok but `touches` overlaps a live lease owned by another), `deferred`/`blocked` (passthrough non-actionable); `done`/`verified` lanes are excluded.
4. `next_action` is the text of the first unchecked `- [ ]` step in the lane's `## Operations` section (role tag stripped), or `null` when every box is ticked or the spec has no checkbox Operations (legacy numbered lists yield `null`, not an error).
5. `hat` is the `(role)` prefix of that first-unchecked step, defaulting to `developer`; `null` when `next_action` is `null`.
6. `collides_with` names the task id of the live lease whose `touches` overlap this lane's (path-segment-prefix algorithm reused from `openup-claims.py`), else `null`; `depends_ok` is the boolean "all `depends-on` satisfied (`done`/`verified`)".
7. `openup-board.py` supports a `top` query that prints the single top pickable lane (`state == ready`, no lease, `depends_ok`, `collides_with == null`) and exits cleanly with a reason when none exists; `refresh` (default) regenerates and prints the board.
8. The board reuses `openup-claims.py`'s frontmatter parsing, dependency resolution, and collision logic (imported, not re-implemented) so the board and pre-flight agree by construction.
9. `.claude/skills/openup-workflow/next/SKILL.md` defines `/openup-next` as one cycle: refresh board → take top pickable lane (or stop cleanly with the reason) → claim + worktree via `/openup-start-iteration` → self-brief per T-016 and assume the lane's `hat` → execute under the lane's track ceremony → tick Operations boxes + persist (`design.md`, run log, state) → exit via `/openup-complete-task` or `/openup-create-handoff` (named as the only two legal exits) → end the session.
10. `docs-eng-process/templates/task-spec.md` `## Operations` becomes a checkbox list (`- [ ] <step>`) with the optional `(role)` hat-tag convention documented; `/openup-create-task-spec` guidance is updated to author checkbox Operations.
11. `.claude/CLAUDE.openup.md` states the rule *no information needed to continue work may live only in a conversation*, documents `/openup-next` as the sequential continue-loop with its two legal exits, and records that ticking an Operations checkbox is sanctioned progress state.
12. Every `.claude/` edit is mirrored to `docs-eng-process/.claude-templates/` so `scripts/check-claude-sync.sh` exits 0; `scripts/tests/test_openup_board.py` passes.

## Approach

Mechanize the queue, then the loop. `openup-board.py` is a thin deterministic layer: it imports `openup-claims.py` (via `importlib`, since the filename is hyphenated) for frontmatter parsing, dependency satisfaction, the path-segment-prefix collision test, and the claims directory, then adds two new pieces — Operations-checkbox parsing (first unchecked box → `next_action`; optional `(role)` prefix → `hat`) and lane-state classification — and emits the JSON queue, sorted so the single pickable lane is first. `/openup-next` is pure composition: it shells out to the board for the top lane, hands that task id + track to the already-correct `/openup-start-iteration` (which owns pre-flight + worktree + lease + state), lets the role self-brief from the repo (T-016) under the chosen hat, executes one track-scoped cycle, ticks the Operations boxes it completes, and routes to one of the two legal exits. The board is never authored by the model ("if a step is deterministic, the harness does it"); the checkbox-Operations change is what makes `next_action`/`hat` derivable. Policy text in `CLAUDE.openup.md` closes the loop: continuation state must live in the repo, and `/openup-next` is the named entry point.

## Safeguards

Invariants and limits that must hold:
- **Token / size budget.** `openup-board.py` ≤ ~280 lines (thin over imported claims logic); `next/SKILL.md` ≤ ~140 lines; `CLAUDE.openup.md` gains ≤ ~12 lines; template/skill edits are localized to the Operations sections.
- **Determinism.** The board invokes no model and uses only the standard library; identical inputs → byte-identical `board.json` (no timestamps, no nondeterminism in the payload).
- **Agreement-by-construction.** The board reuses `openup-claims.py` collision/deps — it must NOT re-implement them; a lane the board calls `ready` must be one `openup-claims.py preflight` would also clear.
- **Reversibility.** New script + new skill + additive policy text are pure additions; the template Operations change and create-task-spec edit are localized; revert by deleting the new files and restoring the numbered Operations block. No change to claims/state schemas.
- **No-go zones.** Do not author `board.json` by hand or from the model. Do not add a hook. Do not change `openup-claims.py`/`openup-state.py` behavior. Do not build T-018–T-021 scope. Legacy numbered-Operations specs must still load (yield `next_action: null`, never crash).
- **Invariant.** `.claude/` and `docs-eng-process/.claude-templates/` byte-identical after the change (`check-claude-sync` green); `.openup/board.json` stays gitignored.
