# T-078 — assess-iteration + milestone-review wiring

## Story

> **As** an agent (or human) driving the OpenUP loop
> **I want** the loop to *converge* at iteration and phase boundaries — verify
> the iteration's evaluation criteria, demo only completed acceptance-tested
> work, feed discovered work back into the roadmap, and at a phase boundary
> **pause for a human go/no-go** instead of silently draining to an empty queue
> **So that** the automation follows OpenUP's iteration-lifecycle Assess-Results
> step and its milestone gates — the safety-critical human decision point the
> framework exists to provide — rather than rolling on unchecked.

INVEST — ✅ Independent (builds on delivered T-075/T-077; T-079 waits on it) · ✅ Negotiable · ✅ Valuable (the convergence + human-gate slice) · ✅ Estimable · ✅ Small (one coherent slice: two resolve paths + one skill + phase-review wiring + schema bump) · ✅ Testable (hermetic resolve/schema/record tests)

## Requirements

1. **`openup-board.py resolve` emits two new decision paths.** `resolve`'s
   precedence gains `assess-iteration` (after `pick`, when the active iteration's
   committed work items are all done and no assessment is recorded) and
   `milestone-review` (when the Development Case marks the current phase's exit
   criteria met, the assessment is recorded, and no milestone decision record
   exists for the current `phase`+`cycle`). `plan-iteration` and `noop` keep their
   meaning; the call stays read-only and always exit 0.
   - **Given** an active iteration whose committed work items are all `done` and
     whose iteration-plan instance has no `## Assessment` section **When**
     `openup-board.py resolve` runs **Then** it returns `path: "assess-iteration"`
     with the active iteration id in `.lane`/`.active_iteration`.
   - **Given** a recorded assessment and a Development Case whose current-phase
     exit criteria `openup-lifecycle.py status` reports as met, with no
     `docs/product/milestones/<phase>-<cycle>.md` record **When** `resolve` runs
     **Then** it returns `path: "milestone-review"` with the derived `phase`/`cycle`.
   - **Given** work items still pending in the active iteration **When** `resolve`
     runs **Then** it returns `resume`/`pick` as today — neither new path fires.

2. **New `/openup-assess-iteration` procedure runs Assess Results.** It checks the
   iteration-plan instance's evaluation criteria, demos **only completed
   acceptance-tested work**, feeds discovered work / defects back into the roadmap
   as new pending rows, writes an `## Assessment` section into the iteration-plan
   instance (through the iteration-plan skill, not by hand), and — when the
   Development Case says the phase is done — names `/openup-phase-review` as the
   next move (it does not itself advance the phase).
   - **Given** an iteration with met evaluation criteria and one incomplete
     (not acceptance-tested) work item **When** `/openup-assess-iteration` runs
     **Then** the incomplete item is excluded from the demo and re-queued as a
     roadmap row, and the `## Assessment` section records criteria pass/fail plus
     the demo scope.
   - **Given** the Development Case's current-phase exit criteria are met after
     assessment **When** the assessment completes **Then** it outputs the
     `milestone-review` next-move (does not write the milestone record itself).

3. **`/openup-phase-review` is wired into the loop and pauses for a human.** On the
   `milestone-review` path it raises the go/no-go through `/openup-request-input`
   (T-074), suspending the lane; on an answered **GO** it writes
   `docs/product/milestones/<phase>-<cycle>.md` (the T-075 decision record) and the
   derived phase advances *because that record now exists* — the skill never sets
   `phase` directly. A **NO-GO** records the decision and re-queues Construction
   work in `cycle`+1 without pretending the product never shipped.
   - **Given** the `milestone-review` path and no answer yet **When**
     `/openup-phase-review` runs **Then** an input-request is created, the lane is
     suspended (`awaiting-input`), and no milestone record is written.
   - **Given** an answered input-request with decision GO **When** the loop resumes
     **Then** `docs/product/milestones/<phase>-<cycle>.md` is written and
     `openup-lifecycle.py status` derives the next phase from it.

4. **`openup-next` handles both new paths and the full `plan-iteration`.** The
   procedure branches on `assess-iteration` (→ run `/openup-assess-iteration`) and
   `milestone-review` (→ run `/openup-phase-review`, pause), and the T-077
   transition note ("treat `plan-iteration` exactly like the old promote") is
   replaced with real Plan-Iteration delegation (call `/openup-start-iteration`
   with no `task_id` so §0b generates the phase-appropriate iteration; the quick
   archetype still degenerates to the single-work-item promote).
   - **Given** `resolve` returns `assess-iteration` **When** `/openup-next` runs one
     cycle **Then** it runs the assessment procedure and exits `ADVANCED` (or
     `DONE` if that produced a milestone-review pause).
   - **Given** `resolve` returns `milestone-review` **When** `/openup-next` runs
     **Then** it pauses via `/openup-phase-review` and exits `DONE` with a
     "milestone-review pending human input" reason.

5. **`.openup/state.json` becomes schema 2.** `schema` const 1→2; add
   `iteration_id` (nullable pointer to the iteration-plan instance) and `cycle`
   (integer, from `openup-lifecycle.py`); `phase` is documented as a **derived
   cache** stamped from lifecycle status, not hand-set. `openup-state.py` validates
   schema 2 and **auto-migrates** a schema-1 file on read (no lane breaks on the
   bump).
   - **Given** a fresh iteration started after this change **When**
     `openup-state.py get` reads state **Then** it validates against schema 2 and
     carries `iteration_id` + `cycle`.
   - **Given** an existing schema-1 `.openup/state.json` **When** `openup-state.py`
     reads it **Then** it migrates in place (adds `iteration_id: null`, derives
     `cycle`, keeps `phase`) and does not error.

6. **The DONE sentinel carries the new meaning.** `openup-next`'s `DONE` reason
   changes from "roadmap empty" to reflect "milestone-review pending human input"
   or "phase milestone accepted / product done" — an outer loop still stops on
   `DONE`, but the reason is now a phase-gate signal, not a drained-queue signal.
   - **Given** a milestone-review pause **When** `/openup-next` exits **Then** the
     last line is `OPENUP-NEXT: DONE — milestone-review pending human input …`.

7. **Every new/changed path is covered by hermetic tests.** Add tests for
   `resolve`'s two new paths (fixture repos: iteration-exhausted, phase-exit-met,
   still-pending), schema-2 validation + schema-1→2 migration, and the milestone
   record write on GO. The existing suite stays green.
   - **Given** the new test module **When** the suite runs **Then** the two resolve
     paths, migration, and record-write assertions pass and no prior test regresses.

## Approach

Keep the drivers dumb and the decisions derived. `openup-board.py resolve` stays
the single read-only decision point: it grows two paths computed from facts
already available — the iteration-plan instance's committed-work-item status
(iteration exhausted?) and `openup-lifecycle.py status` + the Development Case
(phase exit met?). The generative work (judging evaluation criteria, demoing,
writing the assessment) lives in the new `/openup-assess-iteration` procedure;
the human gate lives in `/openup-phase-review`, which only ever *prepares
evidence and pauses* via the existing T-074 input-request machinery and *records*
the human answer — it never advances a phase. State schema 2 names the iteration
and cycle once so "phase" stops being a hand-set label and becomes a cache of the
derived truth. Every deterministic step is a script call; every human decision is
a milestone record.

## Safeguards

- **Derive, never author mid-loop.** Phase/cycle/criteria come from
  `openup-lifecycle.py` + the Development Case, never model judgment inside
  `resolve`. `resolve` stays read-only and exit 0.
- **Humans own the go/no-go.** `/openup-phase-review` only prepares evidence,
  pauses, and records; it never sets `phase`. The derived phase flips only when a
  milestone record exists.
- **Quick archetype keeps its cost.** In the degenerate single-lane case,
  assess-iteration is auto/low-ceremony — the quick archetype must not gain
  ceremony (the program's token-efficiency guardrail).
- **Reversibility.** Schema-1 state auto-migrates (no destructive rewrite of a
  live lane); the new resolve paths are additive (existing paths unchanged).
- **No-go zones.** The micro-increment layer (lanes/leases/fence) and parallel
  iterations (T-079) must not change.
- **Base is `harness-optional`.** Branch off and fence against it
  (`openup-fence.py check --base harness-optional`); do not run `complete-task`
  verbatim (it hardcodes `main`).
