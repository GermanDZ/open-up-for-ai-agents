# T-122 — Process-gate fixes (B8, B9)

## Story

> **As** a lane (agent or human) whose work legitimately regenerates the
> `docs/INDEX.md` trace view, or that begins a session against a prose-style
> roadmap
> **I want** the write-fence to auto-allow a fresh-base INDEX.md regeneration and
> the claim pre-flight's roadmap status to read `## T-NNN:` sections
> **So that** neither the fence nor `session begin` forces a revert-rework or a
> hand-replicated flow around a gap the tools already handle for the other
> roadmap shape

INVEST — ✅ Independent · ✅ Negotiable · ✅ Valuable · ✅ Estimable · ✅ Small · ✅ Testable

## Analysis Context

- **Domain.** Two process-gate defects found in the 2026-07-16
  orchestration-economics review
  (`docs/explorations/2026-07-16-cycle-orchestration-economics.md`, B8–B9). Both
  are self-logged **rework episodes** on the my-product-2 sample, not
  speculative: B8 caused the T-003 revert; B9 forced the T-009 session flow to be
  hand-replicated. Neither touches the LLM path — both are deterministic-script
  correctness.
- **B8 — INDEX.md is a derived view the fence omits.** `docs/INDEX.md` is
  regenerated by `scripts/docs-index.py` and carries the same never-hand-edit /
  regenerate-against-fresh-trunk rule as `docs/roadmap.md` and
  `docs/project-status.md` (stated in `CLAUDE.openup.md` and the derive-don't-edit
  model). But `openup-fence.py`'s `VIEW_PATHS` lists only the two sync-status
  views, so a lane that correctly regenerates INDEX.md is flagged **OUT OF LANE**
  and reverts it — exactly what happened at T-003.
- **B9 — `roadmap_status()` only parses tables.** `openup-claims.py roadmap_status()`
  scans `|`-delimited table rows only. `openup-roadmap.py` already parses **both**
  shapes (table rows **and** prose `## T-NNN:` sections, via `parse_roadmap`). On a
  prose roadmap (the shape this repo's own `docs/roadmap.md` uses below its
  maintenance table), `roadmap_status()` returns `None`, so the archived-dep and
  `session begin` pre-flight paths (`dep_satisfied`) lose their roadmap
  fallback — which is why the T-009 session flow had to be hand-replicated.
- **Scope boundaries.** Process-gate correctness only. No gate-frequency change
  (that is T-123), no tool-surface change (T-121), no contextualization change
  (T-120). No new fence view class, no roadmap-format enforcement change.
- **Definition of done.** B8: a fresh-base INDEX.md regeneration passes the fence
  and a stale-base one is flagged STALE VIEW (not OUT OF LANE). B9:
  `roadmap_status()` returns the correct status for a prose `## T-NNN:` entry and
  keeps working for table rows — sharing `openup-roadmap.py`'s parser so the two
  never drift again. Each has a hermetic regression test.

## Requirements

1. **B8 — `docs/INDEX.md` is fenced as a derived view.** The write-fence treats
   `docs/INDEX.md` exactly like the other class-3 shared views: allowed when the
   base is fresh (or `--allow-views`), flagged STALE VIEW otherwise — never OUT OF
   LANE.
   - **Given** a lane whose diff includes a regenerated `docs/INDEX.md` and whose
     base is an ancestor of HEAD **When** `openup-fence.py check` runs **Then** it
     passes (INDEX.md is not reported OUT OF LANE).
   - **Given** the same INDEX.md change on a **stale** base **When** the fence runs
     **Then** it is reported as a **STALE VIEW** (not OUT OF LANE) and the run
     fails, and `--allow-views` overrides it exactly as for the other views.
   - **Given** `openup-fence.py allowed` **When** it prints the resolved views
     **Then** `docs/INDEX.md` is listed among `views`.

2. **B9 — `roadmap_status()` reads prose `## T-NNN:` sections.** The claim
   pre-flight's roadmap status resolves a task's status from either roadmap shape,
   using the same parser as `openup-roadmap.py` so the two cannot drift.
   - **Given** a `docs/roadmap.md` containing a prose section
     `## T-050:` … `**Status**: completed (2026-..)` **When**
     `roadmap_status("T-050", root)` is called **Then** it returns `"completed"`.
   - **Given** a table-row roadmap with a `| T-050 | … | completed | … |` row
     **When** `roadmap_status("T-050", root)` is called **Then** it still returns
     `"completed"` (no regression to the table path).
   - **Given** a task id absent from the roadmap **When** `roadmap_status` is
     called **Then** it returns `None`.
   - **Given** an archived dep whose plan body status is stale but whose roadmap
     entry (in **prose** form) is `completed` **When** `dep_satisfied(dep, root)`
     is evaluated **Then** the dep is reported satisfied via the roadmap fallback
     (the T-009 path now works for prose roadmaps).

## Behavior Delta

**Modified** — `openup-fence.py`: `VIEW_PATHS` gains `docs/INDEX.md` (+ docstring
note). `openup-claims.py`: `roadmap_status()` delegates to `openup-roadmap.py`'s
`parse_roadmap` (loaded lazily via importlib to avoid the roadmap→claims import
cycle), falling back to the legacy inline table scan only if that module cannot be
loaded. `parallel-lanes.md`: the class-3 derived-view row names `docs/INDEX.md`
(generated by `docs-index.py`).

**Added** — n/a (no new flag, view class, or public function).
**Removed** — n/a (the legacy table scan is retained as a defensive fallback).

## Approach

- **B8** `openup-fence.py`: append `"docs/INDEX.md"` to `VIEW_PATHS`. The existing
  view logic (fresh-base allow / STALE VIEW / `--allow-views`) then covers it with
  zero new branching. Update the module docstring's view list. Add a row to the
  parallel-lanes class-3 table.
- **B9** `openup-claims.py`: add a lazily-cached importlib loader for
  `openup-roadmap.py` (the same file-path importlib idiom the fence uses to load
  claims, and that roadmap.py already uses to load claims — so no module-init
  cycle: the import is deferred to call time). `roadmap_status()` reads the roadmap
  text, runs `parse_roadmap`, and returns the matching entry's already-normalized
  bare-word status (`completed`/`pending`/…) by **exact** id match (`entry["id"]
  == task_id` — stricter and correct vs the old substring match that could match
  `T-1` inside `T-12`). If the parser module cannot be loaded, fall back to the
  legacy table-only scan so the table path never regresses.

## Safeguards

- **No behavior change for table roadmaps.** `parse_roadmap` handles table rows
  identically; the legacy inline scan remains as a fallback, so the table path
  cannot regress even if the loader fails.
- **`openup-roadmap.py` is reused, not modified.** B9 changes only the consumer
  (`openup-claims.py`), so `openup-roadmap.py`'s own tests are untouched.
- **No import cycle.** roadmap.py imports claims.py at module load; claims.py
  loads roadmap.py **lazily at call time** (deferred), so both modules' top-level
  code has already run before either reaches across — no initialization cycle.
- **Fence change is additive.** Adding INDEX.md to `VIEW_PATHS` only *relaxes* the
  fence for a file that was previously always OUT OF LANE; it never tightens any
  existing allowance. Stale-base INDEX is still blocked (as a STALE VIEW).
- **Reversibility.** B8 and B9 are independent; reverting one does not affect the
  other.
