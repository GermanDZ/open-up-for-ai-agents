# T-024 — Write-fence + derived shared views

## Story

> **As a** maintainer running multiple concurrent lanes (agent sessions and humans, each in
> its own worktree/branch)
> **I want** every lane's diff mechanically fenced to its claimed surface, and the shared
> process views (`docs/roadmap.md` Status cells, `docs/project-status.md`) regenerated
> deterministically against a fresh trunk from lane-owned inputs
> **So that** parallel PRs stop conflicting in the roadmap, status, and notes — and when a
> conflict does appear, the resolution is "rebase + re-run the script", never a hand-merge.

INVEST check:
✅ Independent (builds on T-009 claims / T-017 board, changes neither) · ✅ Negotiable ·
✅ Valuable (recurring parallel-PR conflicts in process docs) · ✅ Estimable (1 script + 1
hook + 1 generator change + docs) · ✅ Small (no architecture change) · ✅ Testable
(hermetic CLI tests for fence + assembly)

## Context

`docs/explorations/2026-06-12-multi-worktree-coordination.md` classified every file a lane
writes into four classes (lane-owned / append-only / derived view / global prose) and traced
the remaining parallel-PR conflicts to class-3 content (the views) being hand-written on
every branch against a stale trunk, plus the shared `## Notes` insertion point.

## Behavior Delta

- **Added** — `scripts/openup-fence.py`: deterministic lane-diff check (imports
  `openup-claims.py` parsing/path-match; fresh-base rule for views).
- **Added** — `.githooks/pre-push`: runs the fence for humans when a lane is in flight.
- **Added** — `docs/status-notes/` sharded completion notes; `docs-eng-process/parallel-lanes.md`.
- **Modified** — `scripts/sync-status.py` (vs T-006 behavior): `completed` roadmap cells are
  date-stamped `completed (YYYY-MM-DD)` idempotently; the project-status `## Notes` section
  is assembled newest-first from `docs/status-notes/*.md` when that directory is non-empty.
- **Modified** — `/openup-complete-task` (vs T-017-era steps 3–4): scribe hand-edits of the
  two views replaced by BLOCKING rebase + fence, a sharded note file, and one
  `sync-status.py` run.
- **Removed** — the "prepend an entry to `## Notes`" convention (its content moved to
  sharded files; the section body is now generated).

## Requirements

1. A lane's committed diff outside its surface is detected and blocks completion/push.
   - **Given** a lane branch for T-NNN whose diff touches a file not in its `touches`,
     change folder, or a lane-owned audit tree **When** `openup-fence.py check` runs
     **Then** it exits 8 and prints `OUT OF LANE: <file>` with the remediation.
2. Shared views regenerated against a stale trunk are detected.
   - **Given** a lane branch that modified `docs/roadmap.md` while the base ref is not an
     ancestor of HEAD **When** `openup-fence.py check` runs without `--allow-views`
     **Then** it exits 8 and prints `STALE VIEW: docs/roadmap.md` with a rebase hint.
3. The fence and the claim pre-flight cannot disagree about a task's surface.
   - **Given** a live claim whose `touches` differ from the plan frontmatter **When** the
     fence resolves the allowlist **Then** the claim's `touches` win (T-009 D7), via the
     imported `openup-claims.py` implementation.
4. Completion notes are lane-owned and the Notes view is derived.
   - **Given** files in `docs/status-notes/` **When** `sync-status.py` runs **Then** the
     `## Notes` body of `docs/project-status.md` equals those files' content newest-first
     by filename, and following sections survive; **Given** the directory is absent
     **When** it runs **Then** the section is untouched (backward compatible).
5. Roadmap completion stamps are deterministic and stable.
   - **Given** all track-required gates met **When** `sync-status.py` runs **Then** the
     task's Status cell reads `completed (YYYY-MM-DD)`; **When** it re-runs **Then** the
     original stamp is preserved (idempotent).
6. Humans get the same guardrail without running skills.
   - **Given** `core.hooksPath=.githooks` and a live `.openup/state.json` **When** a human
     pushes an out-of-lane diff **Then** `.githooks/pre-push` blocks with the fence output
     (bypass: `SKIP_OPENUP_FENCE=1`).

## Scope boundaries

- Does NOT change the lease/claim machinery or the board.
- Does NOT derive the roadmap *tables* from frontmatter (full Option A of the exploration)
  — only the Status cell write and the Notes section became script-owned; promoting whole
  tables to generated views is a candidate follow-up.
- Does NOT add CI; the doc names a required PR check as the optional server-side
  enforcement point.
