# Work-Product Instance Frontmatter (T-034)

Every authored OpenUP **work-product instance** under a project's `docs/` carries
YAML frontmatter that declares **machine-readable traceability**: what type of
work product it is, and how it links into the OpenUP trace web (Vision →
Requirements → Work Items → Iteration Plan → Use Cases → Test Cases). This is the
work-product-tier analogue of the [coordination
frontmatter](coordination-frontmatter.md) that the task tier
(`docs/changes/T-NNN/plan.md`) already carries.

The frontmatter is **validated** by `scripts/check-docs.py` (T-036) against
`scripts/docs-meta.schema.json`, and the valid trace relationships are checked
against the KB-derived `scripts/trace-model.json` (T-035).

## Instance frontmatter ≠ template provenance

This is the easiest thing to confuse, so it is the first thing to state.

- The OpenUP-derived **templates** (`docs-eng-process/templates/vision.md`, …)
  ship frontmatter that describes the *template's KB origin*: `type: Template`,
  `source_url`, `related.workproducts`. That is **provenance of the template**.
  The templates are read-only — never edited by this work.
- An authored **instance** — the vision a project actually writes into
  `docs/product/`, a requirement under `docs/changes/` — carries the frontmatter
  documented here: `type: requirement`, `id`, trace fields. That is **semantics
  of the produced work product**.

`check-docs.py` only validates instances — files whose `type` is one of the spine
work-product types below. Template/other files (`type: Template`, untyped notes)
are skipped, so the two never collide.

## Field set

| Key | Type | Required | Meaning |
|---|---|---|---|
| `type` | enum (below) | **yes** | The work-product type. The only mandatory field. |
| `id` | string | no | Stable, project-unique id (`VIS-001`, `REQ-014`, `TC-031`). Other instances reference it in their trace fields. |
| `title` | string | no | Human-readable title. |
| `status` | enum (below) | no | Lifecycle state of the instance. |
| `traces-from` | list of ids | no | Upstream work products this one satisfies/refines (e.g. a requirement `traces-from` a vision). The **authored** trace direction. |
| `verified-by` | list of ids | no | `test-case` ids that verify this work product, where the type warrants coverage. |
| `traced-by` | list of ids | no | **Derived** inverse of `traces-from`, written by the index generator (T-037). Not hand-written. |
| `owner-role` | string | no | OpenUP role accountable for the work product (`analyst`, `architect`, `tester`). |
| `iteration` | string | no | Iteration/phase, e.g. `construction-2`. |

### `type` enum — the v1 core spine

```
vision · requirement · work-item · iteration-plan · use-case · test-case · decision
```

These align to OpenUP KB work-product slugs. v1 deliberately covers the **core
spine** rather than the full OpenUP work-product set (vetoable at review — see the
program plan's Open Questions). The valid `type → type` trace edges between these
are not arbitrary; they are derived from the vendored KB into `trace-model.json`.

### `status` enum

```
draft → approved → implemented → verified → obsolete
```

Distinct from the task-tier coordination `status` enum
(`proposed → ready → in-progress → done → verified`): that tracks *delivery* of a
change; this tracks *maturity* of a work product.

## Key-naming convention — hyphens, not underscores

Trace keys use **hyphens**: `traces-from`, `verified-by`, `traced-by`,
`owner-role`. This matches the coordination-frontmatter convention and what the
readers expect. `additionalProperties` is `false` in the schema, so a typo'd
`traces_from` (underscore) is caught as an unknown property rather than silently
ignored.

## Example

```yaml
---
type: requirement          # required; the only mandatory field
id: REQ-014                # stable id, project-unique
title: Authenticated checkout
status: approved           # draft | approved | implemented | verified | obsolete
traces-from: [VIS-001]     # upstream work products this satisfies/refines
verified-by: [TC-031]      # test coverage (where the type warrants it)
owner-role: analyst
iteration: construction-2
---
```

`traced-by` is intentionally omitted above — it is the derived inverse, written by
the index generator, not by the author.

## Where the rules live

This is a **pointer doc**, not a rules engine. The authoritative contract is the
schema and the trace model:

- `scripts/docs-meta.schema.json` — the instance frontmatter schema (this doc's
  field set, formalised).
- `scripts/trace-model.json` — the KB-derived valid `type → type` trace edges and
  required-coverage rules (generated by `scripts/build-trace-model.py`).
- `scripts/check-docs.py` — the validator that enforces both.
